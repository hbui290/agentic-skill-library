# Evaluation

Pressure-test the router against these boundaries before release: explicit
Librarian request; specialized file format; tool name only; direct edit;
multi-domain task; no-match retry; unavailable CLI; blocked read; multi-phase
handoff; safety signal versus explicit scope; and reference selection.

For each scenario, confirm the trigger decision, minimum references read,
current-phase CLI evidence, truthful status, quota, and no bypass of integrity
or confirmation boundaries. A tool name alone must not trigger routing; a
specialized deliverable or non-routine domain need can.

Release only after focused contracts, strict verification, and whitespace checks
pass. The router remains on demand: do not add a runtime hook, MCP integration,
or automatic router.
